# 扫描与设备适配

## 1. 目标

统一相机扫码、工业 PDA 扫描头、广播、键盘模拟和厂商 SDK 的输入方式，避免业务页面与设备型号绑定。

## 2. Scanner Adapter

建议接口：

```ts
interface ScannerAdapter {
  scan(options?: ScanOptions): Promise<ScanResult>;
  startContinuous?(listener: ScanListener): Promise<void>;
  stopContinuous?(): Promise<void>;
  getCapabilities(): ScannerCapabilities;
}
```

`ScanResult` 至少包含：

- 原始值。
- 扫码类型。
- 来源。
- 时间。
- 可选设备信息。

## 3. 当前基线

当前实现使用 `uni.scanCode`，支持：

- QR Code。
- Bar Code。
- Data Matrix。

这是通用基线，不代表真实工业扫描头已接入。

## 4. 真实 PDA 接入方式

可能包括：

- 厂商广播事件。
- 键盘模拟输入。
- 原生插件。
- Android Intent。
- 厂商 Java/Kotlin SDK。

所有方式都应被转换为统一 `ScanResult`。

## 5. 页面行为

- 页面发起扫码，不关心实现来源。
- 同一时刻只允许一个活动扫描会话。
- 扫码结果未处理完成前应防止重复触发。
- 连续扫描需要显式开始和停止。
- 页面离开时必须释放监听器。

## 6. 扫码格式

每个业务页面应声明：

- 允许的码制。
- 预期业务类型。
- 长度和前缀规则。
- 是否允许手工输入。
- 是否需要服务端解析。

不要在多个页面复制解析正则，应集中到业务解析器或服务端。

## 7. 权限和错误

错误类型至少包括：

- 用户取消。
- 相机权限缺失。
- 设备不支持。
- SDK 初始化失败。
- 扫码超时。
- 条码格式错误。
- 业务对象不存在。

平台错误和业务校验错误必须分开展示。

## 8. Feedback Adapter

建议统一：

```text
success
warning
error
conflict
scanAccepted
```

不同设备可映射到振动、声音、灯光或 Toast。

## 9. Printer Adapter

未来标签打印不得直接嵌入收货页面。打印适配器应负责：

- 打印任务提交。
- 设备选择。
- 状态查询。
- 失败重试。
- 打印结果关联。

## 10. Fake Adapter

H5 和自动化测试应提供：

- 输入模拟条码。
- 成功、取消、权限拒绝、设备故障。
- 连续扫描。
- 重复码和错误码。

## 11. 验收

- 页面不直接调用 `uni.scanCode`。
- 更换 Scanner Adapter 不修改业务流程。
- 页面离开后无残留监听器。
- 重复快速扫描不会重复提交业务命令。
- 权限缺失时有明确设置指引。

# 认证、权限与工厂范围

- 当前阶段：`P1.5：认证与授权闭环`
- 运行时权威：[P1.5 Mobile 认证与授权运行时基线](P1.5-Mobile认证授权运行时基线.md)
- 后端协议权威：[mom-platform P1.5 S00 PR #15](https://github.com/Chris-co-shi/mom-platform/pull/15)

## 1. 应用访问边界

MOM Mobile 只使用 Public Client `mom-mobile-pda`，只允许具备独立 Mobile Access 授权的 `INTERNAL` 用户。

- PDA 是客户端，不是用户类型。
- `SUPPLIER`、`CUSTOMER` 不允许登录 Mobile。
- Client ID 表示应用，不表示用户。
- 无 `client_secret`，Redirect URI 精确匹配。

## 2. 认证流程

- 使用系统浏览器。
- 使用 Authorization Code + PKCE S256 + OIDC。
- 正式环境优先 HTTPS App Link。
- 不在嵌入式 WebView 中输入账号密码。
- PKCE 一次性事务可承受 App 进程回收。
- 冷启动优先使用 Refresh Token 恢复 Session。

完整流程见 [P1.5 Mobile 认证与授权运行时基线](P1.5-Mobile认证授权运行时基线.md)。

## 3. Token 管理

- Access Token：只存在内存。
- Refresh Token：使用 Android 安全存储 Adapter。
- ID Token：不调用业务 API。
- 不使用普通 `uni.setStorage`、Pinia 持久化、明文文件或日志保存 Token。
- 同一 App 实例使用 Single Flight Refresh。
- 每个业务请求最多自动重试一次。
- 403 不触发 Refresh。
- Refresh 本地替换结果不确定时，不得重复使用旧 Token。

## 4. `/api/iam/me`

正式 Mobile Access Context 来自 `/api/iam/me`：

- 当前用户和 `user_type`。
- Client/应用访问结果。
- Roles 与 Permissions。
- `factory_ids`。
- `party_type`、`party_id`。
- Mobile Access 状态。

客户端不得仅解析 JWT 后自行构造完整权限上下文。

## 5. 权限层级

### 5.1 应用访问

决定用户是否能进入 `mom-mobile-pda`，由 `INTERNAL` + Mobile Access 共同决定。

### 5.2 页面权限

决定是否展示收货、上架、领料、退料、入库、发运等入口。

### 5.3 操作权限

决定是否展示提交、取消、重试、转人工或主管操作。

### 5.4 最终授权

前端页面和按钮只改善体验。Gateway 只执行协议与粗粒度 Client 入口校验；业务服务执行最终 Permission、Factory Scope、Party Scope、对象归属和领域状态校验。

## 6. Factory Scope

1. `/api/iam/me` 提供 `factory_ids`。
2. 当前 Factory 可作为非敏感普通偏好保存。
3. 启动、Refresh 和 `/api/iam/me` 更新后必须重新校验偏好。
4. 请求通过 `X-Factory-Id` 传递当前工作上下文。
5. `X-Factory-Id` 不是授权证明。
6. 服务端决定新业务对象的 `factory_id`。
7. Factory 切换时不得自动执行原 Factory 的待处理命令。

P1.5 不建设 Warehouse、Workshop、Line、Department 等通用 Scope。

## 7. Party Scope

Mobile 当前只允许内部用户，不提供供应商、客户主体切换或外部身份冒充。

- 普通内部会话的 `party_type`、`party_id` 通常为空。
- 离线命令仍必须记录创建时 Party 字段，即使为空，以保持契约完整和防止未来错误继承上下文。
- 内部代办必须是后端受控能力，记录 Actor、代表主体和原因；不由 Mobile 自由构造。

## 8. 账号切换与离线命令

每条离线命令必须绑定：

- 创建用户。
- `sid`。
- Client。
- Factory。
- Party。
- 所需 Permission。
- 创建时间。

旧账号创建的命令不能由新账号 Token 自动执行。主管转交若未来实现，必须是显式后端业务操作并有安全/业务审计，客户端不能静默改写命令归属。

## 9. 会话过期、撤销与 Mobile Access 关闭

客户端应：

1. 停止新的写请求和自动同步。
2. 清理 Access/Refresh Token。
3. 保留尚未提交的可恢复输入和离线命令。
4. 引导系统浏览器登录。
5. 登录后重新加载 `/api/iam/me`。
6. 重新校验用户、Session、Factory、Party 和 Permission 后再处理命令。

## 10. 无权限与对象不可访问

需区分：

- 401：会话恢复或重新登录。
- 403：无 Permission、Mobile Access 关闭或操作不允许，不刷新 Token。
- 404：对象不存在或为防枚举隐藏了越权对象。
- 409：业务冲突、状态变化或结果未知。
- 当前 Factory 无权限。

客户端不得根据 404 推断其他 Factory 或 Party 对象存在。

## 11. 设备丢失

设备丢失处置必须支持：

- 撤销对应用户授权 Session。
- 将 `sid` 加入服务端 revoked 集合。
- 清理本地安全存储中的 Refresh Token。
- 停止后台/自动同步。
- 保留但冻结离线命令，等待受控处置。

## 12. 验收

- 只允许 `INTERNAL` + Mobile Access 登录。
- 系统浏览器、PKCE 和 App Link 回调正确。
- Access Token 不落盘，Refresh Token 不进入普通 Storage。
- 并发 401 只刷新一次，403 不刷新。
- `X-Factory-Id` 不被描述为授权凭证。
- 用户 B 不自动同步用户 A 的命令。
- 直接调用仍由服务端拒绝越权。

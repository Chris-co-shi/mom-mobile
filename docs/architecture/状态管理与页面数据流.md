# 状态管理与页面数据流

## 1. 状态分类

移动端状态分为四类：

1. 会话状态。
2. 服务端查询状态。
3. 页面交互状态。
4. 持久化离线命令状态。

四类状态不得混在一个 Store 中。

## 2. 会话状态

包括：

- 用户身份。
- Access Token 生命周期信息。
- 当前工厂。
- 权限摘要。
- 设备与应用版本。

会话状态由专用 Session Store 管理。

## 3. 服务端查询状态

例如：

- 收货任务。
- 工单需求。
- 容器当前状态。
- 库位信息。
- 发运单明细。

原则：

- 服务端始终是权威来源。
- 页面可短暂缓存，但必须明确刷新条件。
- 离线缓存不能冒充实时状态。

## 4. 页面交互状态

例如：

- 当前扫描步骤。
- 已扫描值。
- 表单输入。
- 确认弹窗。
- 提交中。
- 设备错误。

页面状态通常随页面销毁，不应长期持久化。

对于多步骤流程，可使用局部 Flow Store，但应在完成或取消后清理。

## 5. 离线命令状态

由 Offline Repository 和 Service 管理，不放入普通 Pinia Store 作为唯一来源。

页面通过查询服务获得命令 View Model。

## 6. 数据流

### 在线查询

```text
Page
→ API Client
→ Gateway
→ Response DTO
→ Mapper
→ View Model
→ Page
```

### 在线写命令

```text
Page Input
→ Command Builder
→ idempotencyKey
→ API Client
→ Result / Conflict / Unknown
→ Page State
```

### 离线写命令

```text
Page Input
→ Command Builder
→ Offline Service
→ Persistent Repository
→ Queue View
→ Sync Worker
→ API Client
```

## 7. View Model

页面不应直接使用所有 API 字段。建议将 DTO 转换为：

- TaskListItemVM。
- ReceiptSummaryVM。
- ContainerVM。
- LocationVM。
- CommandStatusVM。
- ConflictComparisonVM。

View Model 可包含格式化文本和页面动作可用性，但不能成为业务权威判断。

## 8. 页面恢复

页面因应用切后台、系统回收或路由离开后：

- 未提交表单是否恢复应按业务风险定义。
- 已进入离线队列的命令由队列恢复。
- 已在线提交但结果未知的命令必须通过关联 ID 查询。
- 不应仅依赖内存 Flag 判断是否已提交。

## 9. 网络状态

网络 Store 只表示设备连接状态，不代表 Gateway、认证或后端服务可用。

同步前仍需：

- 健康检查。
- Token 有效性检查。
- 工厂上下文检查。

## 10. 验收

- 普通 Store 清空后离线命令不丢失。
- 页面重建后可通过命令 ID 恢复状态。
- API DTO 变化通过 Mapper 隔离。
- 网络在线但 Gateway 不可用时不会盲目同步。
- 账号切换后会话状态和本地命令边界清晰。

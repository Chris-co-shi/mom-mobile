# P1.5：Mobile 认证与授权运行时基线

- 状态：**Current / Implemented / Completed**
- 原始设计 Slice：`P1.5 S00`；实施状态：`S11、S12 Completed / Merged`
- 后端协议权威：[mom-platform P1.5 S00 PR #15](https://github.com/Chris-co-shi/mom-platform/pull/15)
- Web 对齐参考：[mom-web P1.5 S00 PR #3](https://github.com/Chris-co-shi/mom-web/pull/3)
- 本仓库职责：Android PDA 运行时实现细节

> 本文不得重新定义用户、Token、Claims、Session、RBAC、Factory/Party Scope、Gateway 或业务服务最终授权语义；这些内容以 `mom-platform` 权威设计为准。

## 1. 当前真实状态

本基线已完成 TypeScript 运行时实现：系统浏览器 OAuth/OIDC、PKCE/App Link 回调校验、Access Token 内存、Android Keystore Adapter Contract、Single Flight Refresh、冷启动 Session 恢复、Gateway `/api/iam/me` 与离线身份归属均已交付。

Android Keystore 原生实现、HTTPS App Link 域名关联和真机强杀恢复作为 Phase 02 Mobile 正式联调门禁，不影响 P1.5 Completed / Merged。

## 2. 用户与 Client 边界

Mobile 唯一 Public Client：

- Client ID：`mom-mobile-pda`
- 允许用户类型：`INTERNAL`
- 附加条件：用户必须具有独立 Mobile Access 授权

规则：

1. PDA 是客户端，不是用户类型。
2. `SUPPLIER`、`CUSTOMER` 不允许登录 MOM Mobile。
3. Client ID 表示应用，不表示用户。
4. 无 `client_secret`，Redirect URI 精确匹配。
5. Mobile Access 关闭时必须撤销相关 Session。

## 3. 登录协议

MOM Mobile 统一采用：

- 系统浏览器登录
- Authorization Code
- PKCE `S256`
- OpenID Connect
- 正式环境优先 HTTPS App Link 回调

禁止在嵌入式 WebView 中输入账号密码。密码只提交 IAM 登录页面。

自定义 Scheme 只能作为受控开发或兼容性后备方案，不能替代正式环境优先的已验证 HTTPS App Link。

## 4. 登录流程

```text
App 发起登录
→ 生成 state / nonce / code_verifier / code_challenge
→ 持久化一次性 PKCE 事务
→ 打开系统浏览器
→ IAM 登录与授权
→ HTTPS App Link 返回 App
→ 校验 state / nonce / Client / Redirect URI
→ Code + code_verifier 换 Token
→ 原子替换本地 Refresh Token
→ 清理一次性 PKCE 事务
→ /api/iam/me
→ 校验 INTERNAL + Mobile Access
→ 建立内存 Access Context
```

PKCE 临时事务必须承受 App 进入后台或进程被系统回收。

## 5. Token 存储

### 5.1 Access Token

- 只保存在内存。
- 不写普通 Storage、Pinia 持久化、日志、离线命令或诊断文件。
- App 进程结束后丢失，冷启动通过 Refresh Token 恢复。

### 5.2 Refresh Token

- 使用 Android 平台安全存储能力。
- 通过专用 Auth Secure Storage Adapter 访问。
- 不使用普通 `uni.setStorage`、明文文件、SQLite 明文字段或前端硬编码密钥。
- 本地记录只保存当前有效 Refresh Token 及必要的原子替换元数据。
- 退出、Session 撤销、设备丢失或 Mobile Access 关闭时清理。

### 5.3 ID Token

ID Token 不用于调用业务 API，只在运行时保留 OIDC 校验与必要展示所需的最小信息。

## 6. PKCE 一次性事务

允许持久化：

- `state`
- `nonce`
- `code_verifier`
- `returnRoute`
- `client_id`
- Redirect URI 标识
- 创建时间和过期时间

要求：

1. 使用独立命名空间，与业务离线存储隔离。
2. 回调成功、失败、超时或取消后立即清理。
3. 只接受最近一笔有效事务。
4. 进程恢复后必须验证事务过期时间和 App Link 参数。
5. 不把 Authorization Code 或 Token 写入日志。

## 7. 冷启动 Session 恢复

App 冷启动时：

1. 不从普通 Store 恢复 Access Token。
2. 检查 Android 安全存储中的 Refresh Token。
3. 有 Token 时优先发起 Refresh。
4. Refresh 成功后原子替换 Refresh Token、建立内存 Access Token。
5. 调用 `/api/iam/me`，校验 `INTERNAL`、Mobile Access、Permissions、Factory Scope。
6. 重新校验当前 Factory 和待同步离线命令。
7. Refresh 失败、Session 撤销或绝对 12 小时期限到达时进入系统浏览器登录。

## 8. Refresh Rotation 与本地原子替换

1. 同一 App 实例同一时间只允许一个 Refresh 请求。
2. 所有 401 和前后台恢复共享同一个 Single Flight。
3. 服务端返回新 Refresh Token 后，本地必须原子替换旧 Token。
4. 每个业务请求最多自动重试一次。
5. 403 不触发 Refresh。
6. 网络中断导致“服务端是否已轮换”不确定时，旧 Refresh Token 视为不可再次使用。
7. 不确定状态进入重新登录或受控恢复，不重复提交旧 Token。
8. 不实现客户端宽限窗口。

## 9. 前后台恢复

### 进入后台

- 不在日志或系统任务参数中复制 Token。
- 暂停新的高风险同步调度。
- 保留离线命令和可恢复页面草稿。

### 返回前台

- 检查 Access Token 时效和 Session 状态。
- 需要时进入 Single Flight Refresh。
- 重新加载 `/api/iam/me` 的条件由安全策略、长时间后台或高风险变更触发。
- Refresh/Session 无效时停止写操作并登录。

## 10. `/api/iam/me` 与 Mobile Access Context

Mobile 的正式权限上下文来自 `/api/iam/me`，至少包含：

- 用户标识与显示信息
- `user_type = INTERNAL`
- Client/应用访问结果
- Roles
- Permissions
- `factory_ids`
- `party_type`、`party_id`（内部用户通常为空；代办上下文由后端明确提供）
- Mobile Access 状态

客户端不得只解析 JWT 后自行构造完整权限上下文。

## 11. 当前 Factory

1. 当前 Factory 可作为非敏感普通偏好保存。
2. 启动、Refresh、`/api/iam/me` 更新后必须重新校验。
3. 请求通过 `X-Factory-Id` 携带当前工作上下文。
4. `X-Factory-Id` 不是授权证明。
5. 创建业务对象的 Factory 由服务端上下文确定。
6. 切换 Factory 时不得自动同步原 Factory 的待处理命令。

P1.5 不建设 Warehouse、Workshop、Line、Department 等通用 Mobile Scope。

## 12. Party 与内部代办

MOM Mobile 仅允许 `INTERNAL` 用户。Mobile 不提供 Supplier/Customer Party 切换或外部身份冒充。

若未来允许内部代办场景，必须由 MOM Admin/后端建立受控代表上下文，并记录 Actor、Party 和原因；Mobile S00 不将其作为默认能力。

## 13. Mobile Access 控制

- 页面、任务和操作根据 Permission 改善体验。
- Mobile Access 是进入 `mom-mobile-pda` 的独立授权开关。
- Mobile Access 关闭必须撤销相关 Session。
- 前端隐藏按钮不是最终授权。
- Gateway 只做协议和 Client 粗粒度入口，业务服务负责最终 Permission、Factory/Party、对象归属和领域状态。

## 14. HTTP 状态处理

### 401

- 进入 Single Flight Refresh。
- 每个业务请求最多自动重试一次。
- 恢复失败后停止写操作并登录。

### 403

- 不刷新 Token。
- 显示无 Permission、Mobile Access 关闭或操作不允许。

### 404

- 显示对象不存在或不可访问。
- 不推断其他 Factory/Party 对象是否存在。

### 409

- 进入冲突、状态变化或结果未知处理。
- 离线命令不盲目重试。

### 429

- 遵循 `Retry-After`。
- 不立即放大重试。

## 15. 离线命令身份归属

每条离线命令必须记录：

- 创建用户 `user_id`
- 创建 Session `sid`
- Client `mom-mobile-pda`
- `factory_id`
- `party_type`、`party_id`
- 所需 Permission Code
- 创建时间
- 幂等键、Correlation ID、Schema 版本和命令类型

Token 不进入命令载荷。

同步前必须验证：

1. 当前用户等于创建用户。
2. 当前 Client 为 `mom-mobile-pda`。
3. 当前 Session/重新认证策略允许处理该命令。
4. 当前用户仍有该 Factory 和 Permission。
5. Party 归属与命令创建时一致。
6. 服务端仍执行最终授权和幂等校验。

用户 A 的离线命令不能由用户 B 自动同步。主管转交若未来实现，必须是显式后端业务能力并有审计，不得客户端静默改写归属。

## 16. H5 安全限制

H5 仅用于开发调试、CI、原型评审和 Fake Adapter 联调：

- 按 Web 内存 Token 模型处理。
- 不持久化真实 Refresh Token。
- 不声称具备 Android 安全存储或 App Link 生产安全性。
- 不使用 H5 构建作为正式 PDA 认证安全验收。
- 使用专用开发 Client/环境和非生产凭证。

## 17. Session 撤销与设备丢失

以下情况必须停止业务写入、清理本地 Token 并重新认证：

- 用户禁用或密码变更
- Client 禁用
- Mobile Access 关闭
- Session 撤销或 `COMPROMISED`
- 设备丢失处置
- Refresh Token 重放
- 12 小时绝对 Session 到期

离线命令不能因此静默删除；应保留归属和状态，等待原用户重新认证、显式取消或受控人工处置。

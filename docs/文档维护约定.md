# 文档维护约定

本文档定义 `mom-mobile` 的固定文档分支、中文命名、原型交付和维护流程。

## 1. 固定文档分支

所有文档相关操作统一在：

```text
agent/complete-chinese-docs
```

包括 README、需求、计划、架构、ADR、用户流程、原型、状态矩阵、API 映射、测试和发布文档。

## 2. 标准流程

```text
同步 main
→ 在固定文档分支修改
→ 检查链接与技术事实
→ 创建或更新 Draft PR
→ CI 验证
→ Squash merge
```

禁止直接在 `main` 修改文档。

## 3. 命名规则

- 文件名中文优先。
- `requirements`、`plans`、`architecture`、`adr` 等一级目录保留英文。
- ADR 使用 `ADR-编号-中文标题.md`。
- 阶段计划保留 `Phase` 或 `VS` 编号。
- 原型图片名称应表达设备、页面与状态，例如 `pda-receiving-offline.png`。

## 4. 原型交付门禁

业务页面进入实现前必须具备：

- 业务目标和操作角色。
- 现场入口与前置条件。
- 正常用户流程。
- 扫码、手输、取消和返回路径。
- 竖屏正常态原型。
- 弱网、离线、冲突、重复、设备错误和人工接管原型。
- 页面状态矩阵。
- 组件映射。
- API、权限、数据范围、幂等和离线规则映射。

## 5. 内容真实性

文档必须区分：

- 已实现并可验证。
- 已有代码骨架但未闭环。
- 已设计但未实现。
- 尚待决策。

不得把 H5 构建通过描述为 Android 真机发布完成，也不得把命令入队骨架描述为完整离线同步已经实现。

## 6. README 风格

根 README 应保持现代化项目首页风格，包含项目定位、徽章、系统图、当前能力、技术基线、快速开始、文档导航和路线图，但不得堆砌营销文案或夸大完成度。

## 7. 文档一致性

- `package.json` 和锁文件是版本事实来源。
- `src/pages.json` 是当前页面注册事实来源。
- `src/platform` 是平台能力边界事实来源。
- `src/offline` 是离线命令当前实现事实来源。
- 关键架构变化必须新增 ADR。
- 同一规则只维护一个权威文档，其他位置使用链接引用。
